# Lavra v0.7.8 Release Notes

**Release date:** TBD

---

## Improvements

### `lavra-work-single` now escalates deferred anti-patterns more clearly

When a recurring anti-pattern is discovered but left out of scope, the single-bead workflow now asks for either a `MUST-CHECK` knowledge entry or a project-level agent rule addition. That makes the warning durable for future agents instead of leaving it only as a one-off deviation note.

### Memory sanitization now uses a dedicated Go helper instead of embedded Python

The local memory sanitizer had outgrown the shell wrapper that hosted it. JSONL parsing, normalized dedupe, anchor extraction, drift validation, audit generation, and status assignment were all living inside an embedded Python heredoc in `memory-sanitize.sh`. That shape worked, but it was the wrong boundary: the hook wrapper had become hard to read, hard to test, and too easy to let drift against the fallback path.

That logic now lives in a small Go helper under `plugins/lavra/hooks/memorysanitize/`. The shell script is back to doing shell-script things: scheduling background work, managing the lock, invoking the helper, and rebuilding the SQLite cache afterward.

### The fallback contract is now explicit

We did not turn Go into a hard requirement for every machine. If `go` is available, Lavra builds a local `.memory-sanitize-go` binary in `.lavra/memory/` and uses the full sanitizer path. If `go` is not available, Lavra falls back to the reduced `jq` sanitizer path.

That means recall still works everywhere, while richer drift validation and audit detail stay in the Go path where they are easier to implement and test.

### Installers and self-heal flows now carry the helper source with the hook

Anywhere Lavra copies `memory-sanitize.sh`, it now also copies the bundled `memorysanitize/` source directory. That includes project installs, global hook installs, and the `check-memory.sh` self-heal path. The helper can build in-place on any machine with Go installed, without depending on the original repo checkout still being present.

### Go validation is part of the release surface now

The Go helper has race-tested unit coverage, vet/lint validation, and release build instructions for darwin/linux amd64/arm64. Release procedures now include building helper artifacts and attaching them to the GitHub release.
