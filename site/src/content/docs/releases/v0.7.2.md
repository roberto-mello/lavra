# Lavra v0.7.2 release notes

**Release date:** TBD

One new configuration option for `/lavra-work`, plus fixes to `memory-capture.sh`, the knowledge base, the OpenCode installer, and shell compatibility. Thanks to [xanharris](https://github.com/xanharris) for two PRs and to Dan for reporting the OpenCode installer issues.

## What's new

### `review_scope` config for `/lavra-work`

By default, `/lavra-work` runs `/lavra-review` on everything — every bead, every wave, every change. That's the right default for most projects. But on large, fast-moving codebases it adds significant time per bead, and not every change warrants full multi-agent review. A one-line config edit to fix a typo in a template shouldn't take the same path as an auth refactor.

The new `workflow.review_scope` option in `.lavra/config/lavra.json` gives you control over when `/lavra-review` fires:

```json
{
  "workflow": {
    "review_scope": "targeted"
  }
}
```

- **`"full"` (default):** `/lavra-review` runs on every bead and every wave. Nothing ships without multi-agent review.
- **`"targeted"`:** `/lavra-review` runs only on beads that are P0/P1 priority, or whose title or description contains architecture or security terms. Everything else gets a self-review only.

If you're not sure which to use, leave it unset. `"full"` is the default and the safer option. `"targeted"` is for teams that want to move faster on low-risk work without skipping review entirely on the things that matter.

## Bug fixes

### `memory-capture.sh`

**Corrupted knowledge entries when chaining commands** ([#7](https://github.com/roberto-mello/lavra/pull/7), thanks xanharris)

When multiple commands were chained in one Bash call (`&&`, `;`, newlines), the hook produced corrupted entries in two ways. First, `BEAD_ID` would pick up the entire shell pipeline after the bead ID — entries in `knowledge.jsonl` had `bead` fields containing things like `my-bead\nbd close my-bead 2>&1 | grep "Closed"`. Second, the greedy `.*` in the body extraction matched to the last `bd comments add` in the string, concatenating multiple comment bodies into one blob with embedded shell operators.

The fix scopes extraction to the first `bd comments add` line that contains a knowledge prefix, then extracts BEAD_ID and COMMENT_BODY from that single line only.

Additional fixes:
- **BEAD_ID sanitization**: explicit `tr -cd 'A-Za-z0-9._-'` strip applied after regex extraction, making the intent visible at the use-site.
- **SCRIPT_DIR unified**: eliminated the `SCRIPT_DIR_EARLY` / `SCRIPT_DIR` split — one definition, used throughout.
- **COMMENT_BODY extraction**: merged two mutually exclusive sed passes into one alternation; added explicit `||` alongside `|`.
- **CONTENT sanitization**: `sanitize_untrusted_content` now applied to CONTENT before writing to `knowledge.jsonl`, not only to the comment body going into agent prompts.

### SQL injection in `kb_search`

`kb_search` in `knowledge-db.sh` was building the SQLite query by interpolating the search string directly. It now uses bound parameters. Small surface, but the knowledge base accepts user-contributed content, so string interpolation there was an actual problem.

### `import-plan.sh` subshell isolation

`PREV_BEAD` was assigned inside `$(...)`, which runs in a subshell. The assignment never made it back to the parent shell. Fixed with process substitution.

### Claude installer silently exiting mid-install ([#6](https://github.com/roberto-mello/lavra/pull/6), thanks xanharris)

Steps 5, 6, and 7 of `install-claude.sh` — the ones that copy commands, agents, and skills — were silently exiting on the first file they tried to copy. The cause: `((CMD_COUNT++))` when the counter starts at `0` evaluates to `0`, which bash treats as a falsy exit status. With `set -euo pipefail` active, the script exited immediately. Users ended up with an install that appeared to succeed but had zero commands, agents, or skills.

Fixed by replacing `((var++))` with `var=$((var + 1))` throughout. The POSIX form always exits 0.

### OpenCode installer fixes (thanks Dan)

Several issues reported by Dan that were making the OpenCode installer fail or produce a broken setup:

- **Missing dependency check**: `jq` was required mid-install but not checked at the start. The installer would fail partway through with a cryptic error. It now checks for `jq` upfront.
- **`bun install` ordering**: The conversion script needs `js-yaml`, but `bun install` wasn't running before the conversion. Fixed for both the OpenCode and Gemini installers.
- **Model selection output swallowed**: The model selection UI wrote to stdout, which was being captured by `$(select_model ...)`. All display text now goes to stderr; only the selected model ID goes to stdout.
- **Model allowlist too strict**: A hardcoded allowlist blocked models with unfamiliar names (Zen, Nemotron, Minimax, etc.). Removed the allowlist — format validation is enough. The selector now shows all available models per tier and lets you choose.
- **Perl-style lookahead in grep**: `grep -E` doesn't support `(?!...)` negative lookahead on macOS or Linux. Replaced with a two-stage grep.

### Test suite

Test 3 was checking `.lavra/memory/.gitignore` for the `session-state.md` entry. `provision-memory.sh` writes to `.lavra/.gitignore`. Wrong file — the test was always going to fail.

## Upgrading

Re-run the installer:

```bash
bunx @lavralabs/lavra@latest
```

No breaking changes to bead data, config files, or existing workflows.
