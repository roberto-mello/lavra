# Lavra v0.7.2 release notes

**Release date:** TBD

This is a hardening release — no new features. The main themes are `/lavra-work` cross-platform reliability, `memory-capture.sh` correctness, and POSIX shell compatibility across installers.

---

## What's new

### `/lavra-work` is now self-contained and works on every platform

`/lavra-work` was previously refactored to delegate to separate skill reference files (`single-bead.md`, `multi-bead.md`, `agent-prompt.md`). The intent was to reduce duplication and improve review enforcement, but it introduced a runtime failure mode: those files live at different paths depending on how Lavra is installed — project install, global install, Gemini CLI, OpenCode, and source-repo dev all put them in different locations. When the agent couldn't read the reference file, it would either halt or proceed with no instructions at all.

All content is now inlined directly into `lavra-work.md`. No file indirection at runtime.

`lavra-work-ralph` and `lavra-work-teams` were updated to reference `/lavra-work` phases by name rather than skill file paths.

---

### `/lavra-review` enforcement language unified

The refactor also introduced inconsistent review language. Under `review_scope: "targeted"`, the single-bead and multi-bead paths used different wording — and the single-bead path ended with "self-review only," which agents could read as a complete substitute for `/lavra-review`.

Both paths now use the same rule:

> `/lavra-review` always runs. `review_scope: "full"` runs it on all changes. `review_scope: "targeted"` runs it only on P0/P1 priority beads or beads with architecture/security content.

No more ambiguity about what "targeted" means or when self-review is sufficient.

## Bug fixes

### `memory-capture.sh`

- **BEAD_ID sanitization**: An explicit `tr -cd 'A-Za-z0-9._-'` strip is now applied after regex extraction, making the sanitization intent visible at the use-site rather than relying solely on the upstream regex match.
- **SCRIPT_DIR unified**: `SCRIPT_DIR_EARLY` and `SCRIPT_DIR` were two separate computations of the same path, introduced when `sanitize-content.sh` needed sourcing before the later guards ran. The source call now happens after the early-exit guards, eliminating the split. One `SCRIPT_DIR`, used throughout.
- **COMMENT_BODY extraction**: Sed passes 2 and 3 were mutually exclusive — a line ends with either a shell operator or a bare closing quote, never both. Merged into a single alternation. Also added `||` as an explicit alternative alongside `|` for clarity (previously handled correctly but implicitly).
- **CONTENT sanitization**: `sanitize_untrusted_content` is now applied to CONTENT before it's written to `knowledge.jsonl`. Previously only applied to the extracted comment body going into agent prompts; the JSONL write was unsanitized.

### SQL injection in `kb_search`

`kb_search` in `knowledge-db.sh` was constructing the SQLite query via string interpolation. It now uses bound parameters (`?` placeholders with `-cmd`), closing an injection vector on the search query.

### `import-plan.sh` subshell isolation

`PREV_BEAD` was being assigned inside a command substitution (`$(...)`), which runs in a subshell and doesn't propagate the value to the parent shell. Replaced with process substitution to fix the isolation.

### POSIX arithmetic compatibility

`((var++))` is a bashism that fails under strict POSIX shells. Replaced throughout the installers and `select-opencode-models.sh` with `var=$((var + 1))`.

### Test suite

Test 3 in `test-features.sh` was checking `.lavra/memory/.gitignore` for the `session-state.md` entry, but `provision-memory.sh` writes to `.lavra/.gitignore`. The test was always reading the wrong file.

## Upgrading

Re-run the installer to pick up the updated hooks:

```bash
bunx @lavralabs/lavra@latest
```

No breaking changes to bead data, config files, or existing workflows.
