# Lavra v0.7.2 release notes

**Release date:** TBD

One new configuration option for `/lavra-work`, plus a batch of fixes to `memory-capture.sh`, the knowledge base, and shell compatibility across installers.

---

## What's new

### `review_scope` config for `/lavra-work`

By default, `/lavra-work` runs `/lavra-review` on everything — every bead, every wave, every change. That's the right default for most projects. But on large, fast-moving codebases it adds significant time per bead, and not every change warrants full multi-agent review. A one-line config edit to fix a typo in a template shouldn't take the same path as an auth refactor.

The new `workflow.review_scope` option in `.lavra/config/lavra.json` gives you control over when `/lavra-review` fires:

```json
{
  "workflow": {
    "review_scope": "targeted"
  }
}
```

- **`"full"` (default):** `/lavra-review` runs on every bead and every wave. Nothing ships without multi-agent review.
- **`"targeted"`:** `/lavra-review` runs only on beads that are P0/P1 priority, or whose title or description contains architecture or security terms. Everything else gets a self-review only.

If you're not sure which to use, leave it unset. `"full"` is the default and the safer option. `"targeted"` is for teams that want to move faster on low-risk work without skipping review entirely on the things that matter.

## Bug fixes

### `memory-capture.sh`

- **BEAD_ID sanitization**: An explicit `tr -cd 'A-Za-z0-9._-'` strip is now applied after regex extraction. The upstream regex was doing most of the work, but the intent wasn't visible at the use-site.
- **SCRIPT_DIR unified**: There were two separate computations of the same path — `SCRIPT_DIR_EARLY` (for sourcing `sanitize-content.sh`) and `SCRIPT_DIR` (for `knowledge-db.sh` later). The split existed because the source call used to run before the early-exit guards. Moving it after the guards eliminated the need for two variables.
- **COMMENT_BODY extraction**: Sed passes 2 and 3 were mutually exclusive — a line ends with either a shell operator or a bare closing quote, not both. Merged into one alternation. Also added `||` as an explicit match alongside `|`; it was handled correctly before, but only by accident.
- **CONTENT sanitization**: `sanitize_untrusted_content` is now applied to CONTENT before writing to `knowledge.jsonl`. It was already applied to the comment body going into agent prompts — the JSONL write was the gap.

### SQL injection in `kb_search`

`kb_search` in `knowledge-db.sh` was building the SQLite query by interpolating the search string directly. It now uses bound parameters. Small surface, but the knowledge base accepts user-contributed content, so string interpolation there was an actual problem.

### `import-plan.sh` subshell isolation

`PREV_BEAD` was assigned inside `$(...)`, which runs in a subshell. The assignment never made it back to the parent shell. Fixed with process substitution.

### POSIX arithmetic compatibility

`((var++))` is a bashism. It works in bash but fails in stricter shells. Replaced throughout the installers and `select-opencode-models.sh` with `var=$((var + 1))`.

### Test suite

Test 3 was checking `.lavra/memory/.gitignore` for the `session-state.md` entry. `provision-memory.sh` writes to `.lavra/.gitignore`. Wrong file — the test was always going to fail.
